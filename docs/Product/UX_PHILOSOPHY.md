# UX Philosophy — Evol OS

## Nossa filosofia

A experiência do usuário é parte do produto.

Uma funcionalidade só é considerada pronta quando o usuário consegue utilizá-la sem treinamento.

---

# Princípio 1 — Zero Training

O usuário deve conseguir concluir uma tarefa sem precisar de treinamento prévio.

Se o usuário precisar perguntar "Como faço isso?", existe uma oportunidade de melhorar a interface.

---

# Princípio 2 — Explique antes de pedir

Nunca solicite uma informação sem explicar:

- o que é;
- por que existe;
- quando será utilizada.

Exemplo:

Peso da pergunta

Quanto maior o peso, maior será a influência desta pergunta no resultado final da avaliação.

---

# Princípio 3 — Linguagem Humana

A interface deve falar como pessoas falam.

Evitar termos técnicos quando houver uma alternativa simples.

Exemplos:

❌ Template

✅ Modelo de avaliação

---

❌ Cycle

✅ Período da avaliação

---

❌ Submit

✅ Enviar avaliação

---

# Princípio 4 — Sempre existe um próximo passo

Nenhuma tela pode terminar sem orientar o usuário.

Toda página deve responder:

"O que devo fazer agora?"

Exemplos:

Template criado.

→ Criar primeira seção.

Primeira seção criada.

→ Criar perguntas.

Template concluído.

→ Criar ciclo.

---

# Princípio 5 — Estados vazios ensinam

Nunca mostrar apenas:

"Nenhum registro encontrado."

Sempre explicar:

- o que está faltando;
- por que aquilo é importante;
- qual botão clicar.

---

# Princípio 6 — Uma ação principal

Toda tela deve possuir uma ação principal claramente destacada.

O usuário nunca deve ficar procurando o próximo botão.

---

# Princípio 7 — Progressive Disclosure

Mostrar apenas o necessário naquele momento.

Recursos avançados aparecem conforme o usuário evolui.

---

# Princípio 8 — Coach Mode

Sempre que possível, o sistema deve ensinar boas práticas de RH.

Exemplos:

💡 Dica do Evol

Evite avaliações com mais de 40 perguntas.

---

💡 Boa prática

Feedbacks específicos geram melhores resultados do que comentários genéricos.

---

# Princípio 9 — Ajuda contextual

Ajuda deve aparecer no momento certo.

Não queremos um manual.

Queremos pequenas explicações exatamente onde elas fazem sentido.

---

# Princípio 10 — Nunca deixar o usuário preso

Após qualquer ação importante, a interface deve oferecer uma continuação natural.

Exemplos:

Avaliação enviada.

→ Voltar para avaliações.

Empresa criada.

→ Convidar colaboradores.

Primeiro departamento criado.

→ Criar cargos.

---

# Princípio 11 — Feedback imediato

Toda ação deve gerar um retorno claro.

Exemplos:

✔ Salvo

⏳ Salvando...

⚠ Erro ao salvar

---

# Princípio 12 — Erros ajudam

Mensagens de erro devem explicar:

- o problema;
- como resolver;
- qual o próximo passo.

Nunca mostrar erros técnicos para o usuário final.

---

# Princípio 13 — Teste da Dona Maria

Antes de aprovar qualquer funcionalidade, perguntar:

"Uma pessoa com pouca familiaridade com tecnologia conseguiria concluir essa tarefa sem ajuda?"

Se a resposta for não, a funcionalidade ainda não está pronta.

---

# Checklist obrigatório

Toda nova tela deve responder:

□ O que é isso?

□ Por que isso existe?

□ O que devo fazer agora?

□ Existe uma ação principal?

□ Existe ajuda contextual?

□ Existe estado vazio inteligente?

□ Existe mensagem de sucesso?

□ Existe mensagem de erro amigável?

□ Existe saída da tela?

□ O usuário consegue concluir a tarefa sem treinamento?

Somente depois desse checklist a funcionalidade pode ser considerada concluída.
